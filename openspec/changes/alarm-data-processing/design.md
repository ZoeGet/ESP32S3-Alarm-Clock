## Context

当前项目已经实现了远程任务同步的基础框架（RemoteTaskSync 模块），能够从服务器获取任务并存储到 Flash。但在实际使用中发现两个关键问题：

1. **数据解析不够完善**：现有的 `fetchTasks()` 函数虽然能解析 JSON，但缺少防重复处理机制，可能导致同一任务被多次处理和存储
2. **缺少本地触发逻辑**：存储闹钟时间后，没有本地比对和触发机制，闹钟无法在指定时间自动响起

项目已具备的技术基础：
- RTC 时钟模块（Clock.h）：提供准确的实时时间
- 闹钟模块（Alarm.h）：提供音频播放功能
- Preferences 库：用于 Flash 存储
- ArduinoJson：JSON 解析能力

## Goals / Non-Goals

**Goals:**
- 实现完善的 JSON 数据解析逻辑，支持遍历数组并过滤 alarm_set 类型任务
- 实现防重复处理机制，通过 task_id 对比避免重复存储
- 使用 Preferences 库将闹钟时间存储到独立的命名空间 "alarm_config"
- 实现基于 RTC 时间的本地闹钟比对逻辑
- 实现防重复触发机制，确保同一分钟内只触发一次
- 与现有 RemoteTaskSync 和 Alarm 模块无缝集成

**Non-Goals:**
- 不支持多个闹钟同时存储（仅支持单个最新闹钟）
- 不实现闹钟的取消或编辑功能
- 不处理时区转换（使用设备本地时间）
- 不实现贪睡功能（snooze）

## Decisions

### 1. 数据存储结构：独立命名空间

**决策**: 使用独立的 Preferences 命名空间 "alarm_config" 存储闹钟数据。

**理由**:
- 与 RemoteTaskSync 的 "alarm_data" 命名空间分离，职责清晰
- 便于管理和调试
- 避免不同模块之间的数据冲突

**替代方案**:
- 复用 RemoteTaskSync 的命名空间：会导致职责不清，已被排除
- 使用单个命名空间存储所有数据：不利于模块化维护

### 2. 防重复处理：task_id 对比

**决策**: 在存储前对比 task_id，只有当新任务的 ID 大于本地记录的 ID 时才处理。

**理由**:
- 简单高效，字符串对比即可
- 避免不必要的 Flash 写入，延长 Flash 寿命
- 防止重复触发闹钟

**替代方案**:
- 使用时间戳对比：需要额外存储时间戳，增加复杂度
- 使用哈希值：增加计算开销，不必要

### 3. 时间格式化：HH:mm 字符串对比

**决策**: 将 RTC 时间格式化为 "HH:mm" 字符串，与存储的时间格式进行直接对比。

**理由**:
- 与服务器返回的时间格式一致，无需转换
- 字符串对比简单直观
- 避免处理秒和时区的复杂性

**替代方案**:
- 使用分钟数对比：需要转换计算，容易出错
- 使用结构化时间对比：代码复杂，不必要

### 4. 防重复触发：布尔标志位

**决策**: 使用布尔变量 `isAlarmed` 作为触发锁，当分钟跳转时重置。

**理由**:
- 实现简单，内存占用小
- 逻辑清晰，易于调试
- 适合单闹钟场景

**替代方案**:
- 记录上次触发时间：需要额外存储，逻辑复杂
- 使用时间窗口：实现复杂，容易出错

### 5. 触发时机：每秒检查

**决策**: 在 `loop()` 中每秒检查一次当前时间是否匹配闹钟时间。

**理由**:
- 与现有 loop 周期一致（已有 1 秒延时）
- 响应及时，误差在可接受范围内
- 不增加额外的系统负担

**替代方案**:
- 使用硬件定时器：过于复杂，不必要
- 降低检查频率：可能导致触发延迟

## Risks / Trade-offs

### [风险] 时间格式不匹配

**描述**: 服务器返回的时间格式可能与本地格式化格式不一致。

**缓解**: 
- 统一使用 "HH:mm" 格式（24 小时制）
- 在解析时进行格式验证

### [风险] 分钟边界触发问题

**描述**: 在分钟切换的边界（如 07:59:59 到 08:00:00）可能触发多次。

**缓解**: 
- 使用 `isAlarmed` 标志位锁定
- 只有在分钟匹配且未触发过时才触发
- 分钟不匹配时重置标志位

### [风险] 设备重启后状态丢失

**描述**: `isAlarmed` 标志位在重启后会丢失，可能导致重复触发。

**缓解**: 
- 重启后检查当前时间是否已错过闹钟时间
- 如果当前时间已超过闹钟时间，不触发并重置标志位

### [风险] 与 RemoteTaskSync 的集成冲突

**描述**: 两个模块可能同时访问 Flash 存储。

**缓解**: 
- 使用不同的命名空间
- Preferences 库本身支持并发访问
- 确保每次操作后正确关闭命名空间

## Migration Plan

1. **创建 AlarmTrigger 模块**: 实现数据处理和触发逻辑
2. **集成到 RemoteTaskSync**: 在 fetchTasks() 成功后调用数据处理函数
3. **集成到 main.cpp**: 在 loop() 中添加触发检查
4. **测试验证**: 通过串口日志验证解析、存储和触发流程
5. **优化调整**: 根据实际情况调整防重复逻辑
