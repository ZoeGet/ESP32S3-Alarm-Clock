## Context

当前项目使用ESP32S3开发板，通过WiFi连接NTP服务器获取网络时间，并在0.96寸OLED屏幕上显示。这种方式需要依赖WiFi网络，在离线环境下无法正常工作。

ESP32S3芯片内置RTC（实时时钟）模块，可以在断电后通过备用电池继续计时。本项目将利用这一硬件特性，实现离线时间显示功能。

## Goals / Non-Goals

**Goals:**
- 使用ESP32S3内置RTC模块替代NTP网络时间同步
- 在0.96寸OLED屏幕上实时显示时间（HH:MM:SS格式）
- 实现时间格式化显示（补零对齐）
- 每秒刷新一次屏幕显示
- 保留LED闪烁功能作为状态指示

**Non-Goals:**
- 不实现日期显示（仅显示时间）
- 不实现闹钟功能
- 不实现时间设置UI（通过代码设置初始时间）
- 不实现WiFi连接功能

## Decisions

### 1. RTC库选择：使用RTClib而非ESP32原生RTC
**决策**：使用RTClib库封装RTC操作
**理由**：
- RTClib提供统一的DateTime对象，便于时间格式化
- 代码可读性更好，抽象层次清晰
- 社区支持好，文档完善

**替代方案**：直接使用ESP32的`struct tm`和`getLocalTime()`
- 缺点：需要更多手动配置，代码复杂度增加

### 2. 时间初始化策略：编译时间作为初始时间
**决策**：首次运行时使用程序编译时间作为RTC初始时间
**理由**：
- 无需手动输入时间，自动化程度高
- 时间误差在可接受范围内（编译到运行通常几分钟）
- 通过宏`__DATE__`和`__TIME__`获取编译时间

**替代方案**：手动硬编码时间
- 缺点：每次修改时间都需要重新编译

### 3. 显示刷新策略：每秒清屏+重写
**决策**：每次更新时先清屏再写入新时间
**理由**：
- 避免字符残留（如59变00时）
- 实现简单，代码清晰
- OLED屏幕支持快速清屏

**替代方案**：局部刷新（只更新变化的数字）
- 缺点：实现复杂，需要跟踪上次显示内容

### 4. 错误处理：初始化失败进入死循环
**决策**：如果RTC或OLED初始化失败，程序进入死循环并输出错误信息
**理由**：
- 硬件故障时避免执行无效代码
- 通过串口/OLED提示错误，便于排查
- 符合嵌入式系统故障处理惯例

## Risks / Trade-offs

**[风险] RTC时间漂移** → 缓解：ESP32S3内置RTC精度约±20ppm，每天误差约1-2秒。如需更高精度，可定期通过NTP校准（未来扩展）。

**[风险] 断电后时间丢失** → 缓解：ESP32S3 RTC在深度睡眠和复位后保持计时，但完全断电后时间丢失。如需保持时间，需外接RTC模块（如DS3231）和备用电池。

**[风险] OLED烧屏** → 缓解：时间显示位置固定，长时间显示相同内容可能导致烧屏。当前设计每秒更新秒数，变化频率足够高，风险较低。

## Migration Plan

1. **添加依赖库**：在platformio.ini中添加RTClib依赖
2. **修改main.cpp**：
   - 删除WiFi、NTP相关代码
   - 删除闹钟功能代码
   - 添加RTC初始化和时间读取代码
   - 保留LED闪烁功能
3. **测试验证**：
   - 验证RTC时间是否正确设置
   - 验证OLED显示是否正常
   - 验证LED闪烁是否正常
