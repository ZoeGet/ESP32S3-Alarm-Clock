## Context

当前 ESP32-S3 设备已具备 WiFi 连接能力和基础闹钟功能，但音频播放仅支持简单的蜂鸣声。用户需要通过服务器下发自定义 MP3 音频作为闹钟铃声。硬件上已选用 MAX98357A I2S 功放模块，需要通过软件实现网络音频流式播放功能。

**约束条件：**
- ESP32-S3 内存有限（PSRAM 约 8MB）
- 音频播放需要实时性，不能阻塞其他任务
- 网络不稳定时需要优雅降级
- 功耗敏感，播放结束后需要快速进入低功耗状态

## Goals / Non-Goals

**Goals:**
- 实现基于 I2S 协议的音频输出驱动（MAX98357A）
- 支持从服务器流式下载并播放 MP3 音频
- 与现有闹钟触发逻辑无缝集成
- 支持播放超时和错误处理
- 内存占用控制在合理范围内

**Non-Goals:**
- 不支持本地 SD 卡音频播放（后续可扩展）
- 不支持音频格式转换（仅支持服务器下发的 MP3）
- 不支持多音频并发播放
- 不支持音频均衡器、音效处理等高级功能

## Decisions

### 1. 音频解码库选择：ESP32-audioI2S

**选择：** 使用 `ESP32-audioI2S` 库（by schreibfaul1）

**理由：**
- 专为 ESP32 优化，支持 PSRAM 缓冲
- 内置 MP3 解码器，无需额外依赖
- 支持流式播放（边下载边播放）
- 支持 I2S 输出，兼容 MAX98357A
- 社区活跃，文档完善

**替代方案：**
- `libhelix-mp3` + 自定义 I2S 驱动：开发成本高，不推荐
- `DFPlayer Mini`（硬件解码）：需要额外硬件成本，不支持流式播放

### 2. I2S 引脚配置

**选择：**
- BCLK (位时钟): GPIO15
- LRC (左右声道): GPIO16
- DIN (数据输入): GPIO7

**理由：**
- 与用户提供的硬件连接一致
- GPIO7/15/16 在 ESP32-S3 上是合法的 I2S 引脚
- 避开 WiFi 天线引脚（GPIO11-17 部分引脚）

### 3. 音频缓冲策略

**选择：** 双缓冲机制（2 个 4KB 缓冲区）

**理由：**
- 一个缓冲区用于网络下载，另一个用于音频播放
- 交替使用，避免播放中断
- 总内存占用约 8KB，可接受

**替代方案：**
- 单缓冲区：简单但容易断音
- 大缓冲区（>16KB）：内存占用过高

### 4. 网络请求方式

**选择：** 使用 `HTTPClient` + `WiFiClientSecure` 流式读取

**理由：**
- 与现有代码风格一致
- 支持 HTTPS 安全连接
- 可以逐字节读取响应流，无需一次性下载
- 内存占用低

### 5. 播放超时处理

**选择：** 30 秒播放超时自动停止

**理由：**
- 避免网络异常时无限等待
- 30 秒足够播放完整的闹钟铃声
- 超时后自动降级为蜂鸣声

### 6. 错误降级策略

**选择：** 播放失败时自动切换到蜂鸣声

**理由：**
- 确保闹钟功能始终可用
- 网络异常或服务器故障时仍能提醒用户
- 提升用户体验

## Risks / Trade-offs

### [风险 1] 网络不稳定导致播放中断

**描述：** WiFi 信号弱或服务器响应慢时，缓冲速度跟不上播放速度

**缓解措施：**
- 双缓冲机制提供一定容错
- 播放失败自动切换到蜂鸣声
- 添加连接超时和重试机制

### [风险 2] 内存不足

**描述：** PSRAM 被其他模块占用时，音频缓冲可能分配失败

**缓解措施：**
- 使用动态内存分配，失败时降级为蜂鸣声
- 播放前检查可用内存
- 控制缓冲区大小（2x4KB）

### [风险 3] 功耗增加

**描述：** 音频播放时电流显著增加（约 200-500mA）

**缓解措施：**
- 限制播放时长（30 秒）
- 播放结束后立即关闭 I2S 和功放
- 使用 D 类功放（MAX98357A 本身功耗较低）

### [风险 4] I2S 时钟精度

**描述：** I2S 时钟不准确导致音频失真或杂音

**缓解措施：**
- 使用 ESP32-S3 内部 PLL 生成精确时钟
- 标准采样率（44100Hz 或 22050Hz）
- 测试验证音频质量

### [风险 5] 音频服务器 URL 配置

**描述：** 服务器音频文件 URL 可能变化或配置错误

**缓解措施：**
- 使用环境变量或配置文件存储基础 URL
- 在 platformio.ini 中添加音频服务器地址配置
- 提供错误提示日志

## Migration Plan

### 部署步骤：

1. **添加依赖库**
   - 在 platformio.ini 中添加 `ESP32-audioI2S`
   - 运行 `pio lib install` 安装依赖

2. **配置 I2S 引脚**
   - 在代码中定义 BCLK、LRC、DIN 引脚
   - 确保与硬件连接一致

3. **创建音频播放模块**
   - 实现 `NetworkAudioPlayer` 类
   - 封装 I2S 初始化、网络请求、播放控制

4. **集成到闹钟系统**
   - 修改 `onAlarmTriggered()` 回调
   - 根据 file_id 选择播放音频或蜂鸣声

5. **测试验证**
   - 测试网络正常播放
   - 测试网络异常降级
   - 测试内存和功耗

### 回滚策略：

如果音频播放功能出现问题：
1. 在 platformio.ini 中禁用音频播放编译选项
2. 重新编译上传，恢复为纯蜂鸣声模式
3. 保留 file_id 字段，后续可重新启用

## Open Questions

1. **音频文件 URL 格式：** 服务器端音频文件的完整 URL 格式是什么？是否需要认证？
2. **音频文件时长：** 服务器提供的音频文件标准时长是多少？（建议 30 秒以内）
3. **并发播放：** 是否需要支持多个闹钟同时触发时的音频优先级？
4. **音量控制：** 是否需要软件音量调节？（MAX98357A 支持硬件音量控制）
